# ChangeLog

## [v0.4.0] - 2026-04-01

### Release Summary

Релиз `v0.4.0` делает переходы между workflow-фазами в `gigacraft` более устойчивыми и лучше фиксируемыми.
Поверх `v0.3.0` расширение теперь сохраняет утверждённые spec и plan artifacts отдельными boundary commits перед переходом к следующей фазе, а `Execution Log` в статус-компаньонах получил локальные timestamp'ы с точностью до минуты, что упрощает resume, аудит и разбор длинных execution-сессий.

### Highlights

#### 1. Утверждённые spec и plan artifacts теперь фиксируются до следующей фазы

- После approval workflow теперь ожидает отдельный commit для written spec перед переходом к planning и для approved plan pack перед переходом к execution.
- Для пользователя это означает более чистые phase boundaries: артефакты принятия решения не растворяются в следующих implementation-изменениях и их проще пересматривать, откатывать и использовать как стабильную точку входа.

#### 2. Resume по длинным планам стал понятнее по времени

- `Execution Log` в `-status.md` теперь требует локальные timestamp'ы с точностью до минуты, а не только голую дату.
- Это помогает быстрее понять, когда именно был завершён шаг, validation или checkpoint commit, особенно если работа шла в несколько заходов или через разные execution mode.

#### 3. Workflow лучше различает phase commits и milestone commits

- Boundary commits для approved artifacts теперь зафиксированы отдельно от milestone commits внутри execution-stage.
- Для пользователя это снижает путаницу между “мы утвердили и зафиксировали артефакт фазы” и “мы завершили рабочий milestone реализации”.

## [v0.3.0] - 2026-04-01

### Release Summary

Релиз `v0.3.0` усиливает дисциплину выполнения планов в `gigacraft`.
Поверх `v0.2.0` workflow стал заметно последовательнее в test-first реализации: execution-stage теперь жёстче учитывает TDD-gates, implementer и planning handoff лучше согласованы с тестовыми ожиданиями, а терминология milestones стала единообразной между planning, execution и проверочными скриптами.

### Highlights

#### 1. Выполнение плана стало более test-first по умолчанию

- Workflow теперь заметно раньше подталкивает execution к тестам и проверкам вместо поздней “допроверки в конце”.
- Для пользователя это означает более предсказуемую реализацию изменений и меньший шанс, что важные проверки вспомнятся уже после большого diff.

#### 2. TDD-guardrails стали явнее в handoff между planning и implementation

- Планирование, implementer-dispatch и execution-stage теперь лучше согласованы вокруг ожидания, что задачи должны проходить через test-driven flow, а не только через code-first реализацию.
- Это уменьшает риск, что один этап говорит “сначала тесты”, а следующий фактически ведёт работу по другой модели.

#### 3. Execution milestones стали называться и проверяться одинаково

- Термины вокруг milestones и checkpoint’ов теперь выровнены между skill-инструкциями и smoke-checks.
- Для пользователя это делает прогресс по плану понятнее: меньше путаницы в том, что считается фазой, milestone boundary и commit checkpoint.

#### 4. Workflow лучше отсекает слабые TDD-паттерны

- В release добавлены дополнительные anti-pattern и gate checks вокруг test-driven поведения.
- Это повышает шанс, что длинные execution flow останутся дисциплинированными и не скатятся в формальное “сначала код, потом что-нибудь проверим”.

## [v0.2.0] - 2026-03-31

### Release Summary

Релиз `v0.2.0` концентрируется на надежности execution-stage в `gigacraft`.
После `v0.1.0` расширение стало заметно устойчивее в реальной работе с планами: статус выполнения теперь ведется детерминированно, milestone boundaries получили явные commit checkpoints, `Serena` / `code-index` переоцениваются в момент старта execution, `brainstorming` перестал преждевременно уходить в review, интерактивные блоки вопросов получили жесткий output contract, а `subagent-driven-development` теперь требует реального запуска сабагентов вместо имитации делегации.

### Highlights

#### 1. План теперь легче продолжать после паузы или нового запуска

- Если выполнение прервалось, workflow заметно лучше понимает, где именно остановился, что уже было сделано и какой шаг должен идти следующим.
- Это снижает риск повторного выполнения уже закрытых задач, потери контекста между сессиями и хаотичного “перечитывания всего чата с нуля”.

#### 2. Execution стал более безопасным на длинных задачах

- На границах milestones workflow теперь ожидает осмысленные commit checkpoints вместо большого неструктурированного diff в конце.
- Для пользователя это означает более предсказуемый прогресс, более удобный rollback и более аккуратные промежуточные результаты по мере движения по плану.

#### 3. Поиск по репозиторию стал адаптивнее к реальной среде

- Workflow больше не полагается вслепую на то, что `Serena` или `code-index` “должны быть доступны”.
- При старте execution проверяется реальная готовность helper’ов, а если они недоступны, workflow явно переключается на локальный fallback вместо тихого деградирования поведения.

#### 4. Brainstorming меньше уходит вперед раньше времени

- Review stage теперь жёстче привязан к уже записанному spec, а не к “идее в обсуждении”.
- Для пользователя это означает более логичный порядок работы: сначала договорились и зафиксировали spec, потом уже запускаются reviewer’ы, а не наоборот.

#### 5. Интерактивные вопросы в brainstorming должны срабатывать стабильнее

- `AskUserQuestion` и `ReviewOptions` теперь жёстче ограничены по формату вывода.
- Это уменьшает риск того, что вместо интерактивного элемента пользователь увидит просто текстовый блок или полусломанный UI-output.

#### 6. Делегированное выполнение стало честнее и прозрачнее

- Если выбран `subagent-driven-development`, workflow теперь должен либо действительно запустить сабагентов, либо честно остановиться и предложить fallback.
- Для пользователя это убирает ложное ощущение, что “делегация пошла”, когда на самом деле всё продолжает делаться одним и тем же агентом inline.
